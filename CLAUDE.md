# ONDA Homepage

## 프로젝트 정의
온다마케팅 공식 홈페이지. 정적 HTML 사이트, GitHub Pages 배포.

## 기술 스택
- **구조**: 정적 HTML/CSS/JS
- **배포**: GitHub Pages
- **브랜치**: feature/gumibear012_20250514_detail_responsive

## 세션 시작 시 필수 확인
1. `WORK_STATE.md`를 읽고 이전 작업 상태 확인
2. `git log -5 --oneline`으로 최근 변경 이력 확인
3. `git status`로 미커밋 변경사항 확인
4. 중단된 작업이 있으면 자동으로 이어서 진행

## 크몽(kmong) 작업 진입 시 필독 (하네스)
사용자가 크몽 관련 지시("크몽", "kmong", "55상품", "gig", "my-gigs" 등)를 하거나
`kmong-crawler/` 디렉토리를 다루는 작업이면:
1. **`kmong-crawler/KMONG_CONTEXT.md`를 반드시 먼저 읽기** — 오늘 세션 누적 규칙, 카테고리 매핑, 트랩 전부 정리됨
2. 관련 메모리 확인 (MEMORY.md의 "크몽 신규 등록 자동화" 섹션 링크)
3. 현재 진행 상태: `new-gigs/55-progress.json`, `new-gigs/55-run-log.json`
4. **지시 수행 전 맥락 파악 완료 확인 후 착수**

## 명령어 제한 (필수 준수 — 위반 시 게이트웨이 전체 멈춤)
- **`sleep` 명령은 절대 30초를 초과할 수 없음**. `sleep 60` 이상 금지.
- 모니터링/대기가 필요하면 텔레그램으로 "N분 후 확인" 보고 후 세션 종료.
- **`process` 도구 사용 금지**. `exec` 도구만 사용할 것. process는 행 걸리는 주요 원인.
- **`exec` 도구 1회 실행 시간 60초 이하**. git push, npm install 같은 긴 명령은 `timeout 60 명령` 으로 감싸기.
- **`exec` 연속 호출 최대 5회**, 그 이후 반드시 텔레그램 중간보고 1회.
- 파일 읽기는 `read` 도구, 수정은 `edit` 도구 사용. `exec`로 cat/sed 대체 금지.
- `git push` 전에 반드시 커밋 먼저. push 실패 시 재시도 1회만, 그래도 실패 시 텔레그램 보고.

## 배포 규칙 (필수 준수)
- **모든 배포는 Cloudflare(Pages/Workers)만 사용. Vercel 사용 절대 금지.**
- **테스트 시안/미리보기** → Cloudflare Pages 배포 + 링크 자동 제공.
- 정적 사이트이므로 `wrangler pages deploy` 직접 사용 가능.
- 배포 후 반드시 접속 확인하고 링크와 함께 결과 보고.

## 작업 규칙
- 작업 시작 전 WORK_STATE.md의 "현재 작업", "진행 상황", "다음 단계" 확인
- 작업 완료/중단 시 반드시 WORK_STATE.md 업데이트 후 git commit
- 커밋 메시지는 한국어로 작업 내용 요약
- 세션 끊김 대비: 중요 진행사항은 즉시 WORK_STATE.md에 기록

## 디버깅/디버깅 루프 정의 (영구)
- 당장 발생한 오류 해결만이 아닌, 파생되는 오류까지 전수 테스트+확인
- 모든 부분의 오류가 0이 될 때까지 반복하는 것이 "디버깅 완료"
- 흐름: 오류 수정 → 빌드 → 파생 오류 확인 → 수정 → 빌드 → ... → 에러 0 = 완료
- 한 곳 수정이 다른 곳에 영향 줄 수 있으므로 반드시 전체 빌드+테스트로 검증
- "에러 하나 고쳤습니다"는 디버깅 완료가 아님. 전체 0 에러가 디버깅 완료

## 작업 완료 규칙
- 모든 작업 완료 시: 보고서 + 개선점 제안(최소 1개) 필수
- "완료했습니다"만 보내기 금지, 구체적 개선점 포함
- 작업완료 = 테스트+디버깅 완전 완료 (위 디버깅 루프 정의 충족)

## 대형 작업 자동 청크 분할 (필수 — 멈춤 방지)
대형 작업을 한번에 처리하면 context/timeout으로 멈춘다. **반드시 아래 규칙을 따를 것.**

### 1단계: 작업 분석 (착수 즉시, 코딩 전)
- 지시를 받으면 **먼저 WORK_STATE.md에 단계별 계획을 작성**
- 각 단계는 **파일 1개 또는 섹션 1개** 수준으로 쪼갤 것
- 예: "UIUX 전체 리디자인" → Step1: header/nav, Step2: hero, Step3: section1, Step4: footer...
- 텔레그램에 "N단계로 나눠서 진행합니다" 보고

### 2단계: 단계별 실행 (한번에 1단계만)
- **한 턴에 파일 1개만 생성/수정** (300줄 이하 권장)
- 한 단계 완료 → 즉시 git commit + WORK_STATE.md 체크표시 업데이트
- 텔레그램에 "Step N/M 완료" 진행 보고
- **절대 다음 단계로 넘어가기 전에 커밋할 것**

### 3단계: 자동 이어가기
- 각 단계 커밋 후 WORK_STATE.md의 "다음 단계"를 보고 자동으로 다음 단계 착수
- 세션이 끊겨도 WORK_STATE.md를 읽으면 어디까지 했는지 바로 알 수 있음
- 모든 단계 완료 시 → 최종 보고 + 배포

### 금지사항
- ❌ 500줄 이상 파일을 한번에 작성
- ❌ 파일 3개 이상을 커밋 없이 연속 수정
- ❌ WORK_STATE.md 업데이트 없이 다음 단계 진행
- ❌ 텔레그램 중간보고 없이 5분 이상 묵묵히 작업

### 병렬 처리
- 독립적 작업 2개 이상이면 coding-agent 서브에이전트 활용 (최대 2개 동시)
- 서브에이전트도 위 청크 규칙 동일 적용

### 텔레그램 회신 규칙
- 작업 착수 즉시 "접수, N단계로 분할 진행" 회신
- 각 단계 완료마다 "Step N/M 완료 + 요약" 보고
- 컨텍스트 관리: 대화가 길어지면 핵심 요약만 유지

## 프로젝트 범위 판단 규칙 (Project Scope Guard)

### 원칙: "내 소관이 아닌 작업은 하지 마라"
이 인스턴스는 **onda-hompage** 프로젝트 전용이다. 다른 프로젝트의 코드를 수정하거나 배포하지 마라.

### 도메인 → 프로젝트 매핑
| 도메인/URL | 프로젝트 | 담당 |
|-----------|---------|------|
| threeway1.com / www.threeway1.com | onda-hompage | 이 인스턴스 (onda_homepage) |
| onda-ad.pages.dev | onda-ad | 000 인스턴스 |

### 배포 대상 확인
- onda-hompage 배포 시 → threeway1.com에 반영되는 CF Pages 프로젝트로 배포
- **onda-ad.pages.dev에 배포하면 threeway1.com에 반영 안 됨** — 절대 혼동 금지
- 배포 완료 보고 전 → 실제 threeway1.com 접속 확인 필수

### 대화 컨텍스트 인식
- 사용자 메시지 내 URL/도메인/키워드 반드시 분석
- "이거", "그거" 지시대명사 → 이전 대화에서 실제 대상 파악
- 다른 인스턴스(000 등) 소관 메시지에 반응 금지
- 모호하면 질문으로 확인 — 추측 처리 금지
