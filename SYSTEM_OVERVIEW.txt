## 2. 데이터 수집 (크롤링)

### 2.1 AdLog 크롤러 (`crawler/adlog.ts`, 1,606줄)
**핵심 데이터 소스.** AdLog(adlog.kr)에서 네이버 플레이스 순위/N2/리뷰 데이터 수집.

**수집 데이터:**
| 필드 | 설명 | 파싱 위치 |
|---|---|---|
| rank | 네이버 플레이스 순위 | cells[0] |
| business_name | 업체명 | cells[1] |
| visitor_review_count | 방문자 리뷰 수 | cells[2] |
| blog_review_count | 블로그 리뷰 수 | cells[3] |
| n1_score | N1 (저품질 판별) | cells[4] |
| n2_score | N2 (핵심 순위 지표) | cells[5] |
| n3_score | N3 (무시) | cells[6] |
| place_id | 네이버 플레이스 고유 ID | `<tr class='tr_PLACEID'>` |

**API 엔드포인트:**
- 로그인: `POST https://adlog.kr/bbs/login_check.php` (mb_id, mb_password)
- 순위 조회: `POST https://adlog.kr/adlog/ajax_place_list.php`
  - 파라미터: api_section=10, api_type=90, place_keyword, master_disp=1, place_date
  - 응답: `{ code: '0000', items: '<HTML table>', place_date: 'YYYY-MM-DD' }`

**구조 변경 자동 감지 (2026-03-11 구현):**
- 헤더 기반 동적 컬럼 매핑 (`buildColumnMapping()`)
- 파싱 성공률 50% 미만 → 구조 변경 감지 알림
- 3개 키워드 연속 실패 → 크롤링 자동 중단 + 텔레그램 긴급 알림
- 저장 전 데이터 검증 (rank ≤ 0, n2=0, 전일 대비 10배 차이 등)

## 3. 데이터베이스 (Supabase)

### 3.1 핵심 테이블

**businesses** — 업체 마스터
```
id, name, place_id (네이버 고유ID), place_url, keyword_id (FK),
is_managed (관리업체 여부), contract_start, contract_end,
is_blacklisted, industry_id, assigned_staff_id
```
- `is_managed=true`: UI에 노출되는 실제 관리업체 (현재 ~25개)
- `is_managed=false`: 크롤링 수집 업체 (87,000+개, 분석용)

**keywords** — 키워드 마스터
```
id, keyword, is_managed, monthly_search_volume, difficulty_grade,
n2_avg_top5, keyword_category, is_paused
```

**daily_snapshots** — 일일 순위 스냅샷 (핵심)
```
keyword_id, business_id, snapshot_date,
rank, n1_score, n2_score, n3_score, n2_change,
blog_review_count, visitor_review_count, save_count,
total_business_count, confidence_score
```
- UNIQUE: (keyword_id, business_id, snapshot_date)
- 매일 ~30,000건 적재

**daily_inputs** — 트래픽 투입 기록
```
business_id, input_date, source_id (FK→sources),
action_type_id (FK→action_types), quantity
```

**contract_tracking** — 계약 관리
```
business_id, contract_start, contract_end,
guarantee_days (25일), achieved_days,
target_rank (5위), status (active/cancelled/completed/extended)
```

**sources** — 매체사
```
id, name (뭉치 플러스, 퍼플페퍼, 말차, 애드앤에이, 애드온, 레이서 등)
```

**action_types** — 액션 유형
```
id, name (유입, 길찾기, 메뉴찾기, 영업시간 등)
```

## 4. 대시보드 UI (관리자)

### 4.1 메인 대시보드 (`app/dashboard/page.tsx`, 633줄)
**URL:** /dashboard

**카드 구성 (업체별):**
- 순위 배지 (1~5위 초록, 6~10위 노랑, 11+ 빨강)
- 업체명 + 키워드
- N2 점수 + 변동
- 블로그/방문자 리뷰 수 + 변동
- 저장수 + 예약 리뷰
- 30일 추이 차트 (N2/순위)
- 기간별 비교 (1일/5일/10일/20일/30일)
- 경쟁사 Top5 테이블 (순위/N2/리뷰)

**상태 판단:**
- 🟢 안정: rank ≤ 5 + N2 유지/상승
- 🟡 정상: rank > 5이지만 추세 상승
- 🟠 주의: rank > 5 + N2 정체
- 🔴 위기: N2 4일 연속 하락 또는 rank > 10

### 4.2 매체관리 (`app/dashboard/businesses/page.tsx`, 1,661줄)
**URL:** /dashboard/businesses

**탭 구성:**
| 탭 | 설명 |
|---|---|
| 전체 | 계약중 + 미계약 (계약종료 제외) |
| 계약중 | active 계약이 있는 관리업체 |
| 미계약 | 관리업체 중 계약 미등록 |
| 계약종료 | status=cancelled/completed + is_managed=false |
| 투입내역 | 트래픽 투입 기록 |

**카드 기능:**
- 현재 순위 + 계약 기간 + 보장일/진행일 표시
- 진행 중인 투입 내역 (매체/액션/타수/기간)
- 수정/중단/연장/투입등록 버튼
- ⚠️ 계약 만료 경고 배지 (만료되어도 자동 이동 안 함)
- 계약종료 버튼 → 확인 모달 → 계약종료 탭 이동
- 재계약 버튼 → 새 계약기간 입력

**업체 등록:**
- place_id URL 파싱 (`/place\/(\d+)/`)
- 중복 체크 (same place_id → 409 에러)
- 비활성 업체 자동 재활성화
- 키워드 중복 생성 방지 (getOrCreateKeyword)

**투입 등록:**
- 매체 선택 (sources 테이블)
- 액션타입 선택 (action_types 테이블)
- 타수 입력 (100타 단위, 퍼플페퍼는 50타)
- 시작일/종료일
- 새 투입 등록 시 기존 활성 투입 자동 종료

## 13. 핵심 비즈니스 로직

### 13.1 N2 지표 체계
- **N1**: 저품질 판별 지표 (참고용)
- **N2**: 핵심 순위 지표 (이것이 올라야 순위가 오름)
- **N3**: 무시

### 13.2 매체 운영 규칙
- 기본 투입: 100타 단위
- 퍼플페퍼: 50타 단위 가능
- 2주 로테이션 필수 (동일 매체 지속 시 작업 감지)
- 업종별 로직 다름 (맛집/술집/카페 완전 별개)

### 13.3 계약 체계
- 30일 단위, 25일 보장
- 5위 이내 보장
- 미달 시 연장
- 계약 만료 시 수동 종료/재계약 확인 (자동 이동 안 함)

### 13.4 투입 자동 종료
- 새 투입 등록 시 같은 업체의 기존 활성 투입 → 종료일을 전날로 변경
- 매체 무관 (다른 매체여도 기존 투입 종료)

### 13.5 데이터 보존
- UI 삭제 = 소프트 삭제 (is_managed=false)
- 하드 삭제 절대 금지
- 모든 이력 데이터 영구 보존
